# Words MVP 项目进度

这个项目当前实现的是一个命令行版英语生词抽取 MVP，目标是先跑通最小闭环：用户提供一段英文 `.txt` 文本，系统从中抽取可能不会的单词，基于 ECDICT 和上下文选择词义，并保存用户对具体义项的掌握状态。

当前已经有一个轻量本地 Web 前端，用于验证核心闭环。它不是完整产品形态，重点是让文本选择、上传、候选词查看和义项状态更新可以在浏览器中操作。

## 当前已实现

### 1. 文本导入与预处理

项目已经可以读取 `.txt` 文件，并完成基础文本清洗、分句、英文 token 提取和词形还原。

预处理阶段会保留句子边界和 token 所在句子位置。这样后续抽出生词时，可以把单词和原文句子绑定起来，而不是只得到一个孤立的单词。

当前分词和词形还原优先使用 spaCy。如果完整英文模型不可用，代码里也保留了较轻量的 fallback 路径，保证基础流程不会完全中断。

### 2. 候选生词抽取

项目已经实现规则评分版的候选词抽取。

抽取逻辑不是简单列出全文所有单词，而是会综合考虑停用词、短词、专有名词、词形合并、文本出现频次、基础词表、目标词库和用户历史状态。

目前的核心思路是：

- 常见停用词和明显基础词会被过滤或降权。
- 同一词的不同形态会尽量合并到 lemma。
- 在文本中重复出现的词会提高优先级。
- 出现在目标词库中的词会提高优先级。
- 用户已经标记为知道、忽略或已加入生词本的词，会影响后续推荐。

当前推荐流程分两阶段。

第一阶段是按 lemma 生成运行时候选词。系统会先过滤掉不适合作为候选的 token：

- lemma 不像英文词的内容会过滤，例如不符合英文单词模式的 token。
- 1 到 2 个字母的短词会过滤，除非它出现在目标词库中。
- spaCy 标记为停用词的词会过滤，除非它出现在目标词库中。
- 内置 fallback 停用词会过滤，例如 `the`、`a`、`is`、`and`、`of`。
- 专有名词会过滤，除非它出现在目标词库中。
- 同一 lemma 会聚合到一个候选词，使用该词在文本中的出现次数作为频次。

过滤后，每个候选词会计算 `unknown_score`，当前公式是：

```text
unknown_score = 词汇难度分 + 目标词库加权 + 文本频次加权 - 基础词扣分 - 用户已知词扣分
```

各项含义如下：

- 词汇难度分：基础词固定较低；非基础词会根据词长增加分数，带有 `tion`、`ment`、`ity`、`ive`、`ous` 等形态特征会有少量加分。
- 目标词库加权：如果 lemma 出现在目标词库中，会增加推荐分。
- 文本频次加权：同一个 lemma 在当前文本中出现越多，分数越高，但有上限。
- 基础词扣分：如果 lemma 在基础词表中，且不在目标词库中，会明显扣分。
- 用户已知词扣分：这是早期 lemma 级接口保留项；当前完整 MVP 主流程主要使用义项级状态过滤。

第二阶段是把候选词映射到具体义项。系统根据 lemma 从 ECDICT 取候选义项，再结合上下文选择 `sense_id`。只有拿到具体义项后，才会读取用户对该义项的状态。

义项状态对推荐的影响如下：

- `new`：用户还没有处理过该义项，可以展示为待处理候选。
- `known`：用户已经掌握该义项，后续待处理候选中会过滤掉这个义项。
- `ignored`：用户暂时忽视该义项，后续待处理候选中会过滤掉这个义项。
- `learning`：用户已加入生词本，后续不再作为待处理生词展示，后续应进入复习队列。
- `archived`：当前实现允许保存该状态，但推荐过滤策略目前主要过滤 `known`、`ignored`、`learning`。

注意：状态过滤是按 `sense_id` 做的，不是按 lemma 做的。因此用户掌握 `charge` 的“收费”义项，不代表掌握 `charge` 的“指控”义项；如果后续上下文被判定为另一个 `sense_id`，仍然可以被推荐。

当前内置的是小型 seed 词表，主要用于跑通流程。后续可以把基础词表替换成 NGSL，把目标词库替换成 CET4/CET6、考研、IELTS、TOEFL 或其他场景词库。

### 3. ECDICT 与上下文释义

项目已经切换为 ECDICT 格式词典作为参考词典。

每个候选词会绑定原文句子，并取相邻句子作为上下文。系统会从 ECDICT 中读取该词的候选义项，再让 DeepSeek V4 Flash 在候选义项中选择最符合上下文的一项。

当前上下文窗口固定为最多 3 句：目标词第一次出现的句子、前一句、后一句。如果目标词出现在文档第一句，就只取当前句和后一句；如果出现在最后一句，就只取前一句和当前句。实现位置是 `src/words_mvp/meanings.py` 里的 `get_candidate_context()`。

ECDICT 查词当前会优先使用候选词在原文中的词形，例如 `revealing` 会先查 ECDICT 的 `revealing` 词条；如果原词形没有命中，再回退到 lemma，例如 `reveal`。这样单词卡片能显示“当前词形”的词性和意思，同时仍然用 lemma 组织同一词族。当前查词时没有先用 token 的词性过滤 ECDICT 结果，因为 spaCy fallback 或不同模型下词性可能不稳定；但 ECDICT 义项里的 `pos` 会被保留下来，并一起传给 DeepSeek 作为选择依据。后续如果词性质量稳定，可以在 DeepSeek 前增加“词性优先过滤，保留 fallback”的策略。

ECDICT 义项解析规则是：读取 `translation`、`definition`、`pos` 等字段；如果中文释义里用分号或中文分号分隔多个意思，会拆成多个候选义项；每个候选义项会生成稳定的 `sense_key`，并保留 `meaning_zh`、`definition_en`、`pos`、`sense_rank`、`source_sense_id`。

单词卡片中的相关形式/派生词不是只依赖 ECDICT。当前顺序是：先保留当前原词和 lemma，再读取 ECDICT 的 `exchange` 字段，然后用 WordNet 的 `derivationally_related_forms()` 查找跨词性派生关系，例如 `purification n.` 对应 `purify v.`，最后再用本地词干规则兜底。WordNet 只负责发现英文相关词，中文释义、词性和常见程度仍然回到 ECDICT 查询；如果某个 WordNet 相关词不在当前 ECDICT 文件中，卡片不会显示它。

传给 DeepSeek 的 system prompt 是：

```text
你是英语词义消歧助手。必须只在给定 ECDICT 候选义项中选择，不要创造新释义。输出严格 JSON。
```

user prompt 是一个 JSON 对象，包含：

- `word`：候选词 lemma。
- `surface`：原文中的词形。
- `context`：前一句、当前句、后一句拼接出的上下文。
- `senses`：ECDICT 候选义项列表，每项包含 `index`、`sense_key`、`meaning_zh`、`definition_en`、`pos`。
- `instruction`：要求只从 `senses` 中选择最符合 `context` 的一个义项，并返回 `{"index": 数字, "confidence": 0到1, "evidence": "中文简短理由"}`。

DeepSeek 请求还设置了 `response_format=json_object`、`temperature=0` 和 `thinking.type=disabled`，目的是让输出稳定且便于解析。

DeepSeek 只负责在 ECDICT 给出的候选义项中选择，不负责自由生成新释义。这样可以降低模型幻觉，把释义边界约束在词典数据内。

如果没有配置 `DEEPSEEK_API_KEY`，或者调用失败，系统会回退到本地规则选择。当前仓库内置的是 ECDICT 兼容格式的 sample CSV，用于跑通流程；正式使用时应替换为完整 ECDICT 数据。

### 4. 用户选择与状态保存

项目已经支持三类用户操作：

- 加入生词本
- 标记为已经知道
- 本次暂时忽视

这些状态会保存到 SQLite。下一次处理文本时，用户历史状态会参与候选词抽取，避免反复推荐用户已经处理过的具体义项。

当前状态影响规则是：

- `known` 会从待处理候选结果中过滤。
- `ignored` 会从候选结果中过滤。
- `learning` 表示已加入生词本，也会从待处理候选结果中过滤，后续应进入复习队列。

当前状态绑定的是 `sense_id`，不是 lemma。也就是说，同一个词的不同意思可以拥有不同学习状态。

### 5. SQLite 数据持久化

项目已经建立 SQLite 存储层，用来保存文档、词条、义项、文本出现位置和用户义项状态。

当前核心表包括：

- `documents`：保存输入文件名、清洗后的文本内容和创建时间。
- `lexemes`：保存词条和词条常见程度排序。
- `word_senses`：保存具体义项，当前来源为 ECDICT。
- `text_occurrences`：保存词在原文中的出现位置、句子和上下文。
- `user_sense_states`：保存用户对具体义项的长期状态。
- `user_sense_events`：保存用户状态变化日志。

候选词列表不再作为独立表持久化。候选词是运行时结果，由当前文本、词表、词频、上下文义项和用户义项状态动态计算。

SQLite 数据库会在首次运行完整 MVP 流程时自动创建。

### 6. 统一配置

项目已经支持通过 YAML 配置统一管理运行参数。

配置文件集中管理输入文本路径、数据库路径、ECDICT 路径、DeepSeek 模型配置、基础词表路径、目标词库路径、候选词数量和最低分数等参数。

命令行参数仍然可以覆盖配置文件中的部分设置，方便临时调试。

### 7. 轻量 Web 前端

项目已经根据 `plans/frontend_plan.md` 增加 FastAPI 前端。

当前前端支持：

- 查看 `input_texts/` 下已有 `.txt` 文件。
- 上传新的 `.txt` 文件到 `input_texts/`。
- 点击运行候选生词抽取。
- 查看候选词、上下文、ECDICT 义项、DeepSeek/fallback 选择方法、`sense_id` 和 `occurrence_id`。
- 对具体义项执行三种操作：加入生词本、已知道、忽视。
- 操作后写入 `user_sense_states` 和 `user_sense_events`。
- 页面上查看数据库基础统计。
- 调试时通过页面按钮清空当前数据库运行数据。
- 进入生词本查看已加入学习的义项，按最近加入顺序倒序排列。
- 点击生词本条目查看单词卡片，卡片显示原词、lemma、当前义项、上下文和 ECDICT 相关词形/派生词；当前加入生词本的具体义项会加粗显示。

前端仍然复用现有 Python 后端逻辑，不在浏览器里重新实现分词、抽词、释义选择或数据库更新。

## 当前处理流程

当前完整流程是：

1. 读取 `.txt` 文件。
2. 清洗文本并分句。
3. 提取英文 token 并做标准化和词形还原。
4. 根据规则抽取候选生词。
5. 为每个候选词找到原文句子和上下文。
6. 从 ECDICT 中读取候选义项。
7. 调用 DeepSeek V4 Flash 从候选义项中选择最符合上下文的一项。
8. 保存文档、词条、义项和文本出现位置到 SQLite。
9. 用户选择加入生词本、已知道或忽视。
10. 用户状态按 `sense_id` 写入 SQLite，并影响后续候选词推荐。

## 当前项目结构

`src/words_mvp/preprocess.py` 负责文本清洗、分句、分词和词形还原。

`src/words_mvp/vocabulary.py` 负责候选生词过滤、聚合和评分。

`src/words_mvp/wordlists.py` 负责加载基础词表和目标词库。

`src/words_mvp/meanings.py` 负责 ECDICT 加载、上下文提取和 DeepSeek 义项选择。

`src/words_mvp/db.py` 负责 SQLite 表结构、数据写入和用户义项状态更新。

`src/words_mvp/pipeline.py` 负责串联完整 MVP 流程。

`scripts/run_mvp.py` 是当前完整流程入口。

`scripts/update_word_status.py` 用于更新用户对某个词的状态。

`web/app.py` 是轻量前端入口，提供页面和 API。

`web/templates/index.html` 是主页面模板。

`web/static/` 放前端样式和少量交互脚本。

`configs/default.yaml` 是当前统一运行配置。

`input_texts/` 用于存放需要提取生词的原始 `.txt` 文件。

`data/wordlists/` 放基础词表和目标词库 seed 文件。

`data/dictionaries/` 放 ECDICT 兼容格式词典数据。

## 当前边界

当前版本仍然是 MVP 验证阶段，有几个明确边界：

- 没有前端页面，当前通过命令行运行。
- DeepSeek 调用依赖 `DEEPSEEK_API_KEY`；没有配置时会回退到本地规则。
- 当前仓库内置的是 ECDICT sample CSV，不是完整 ECDICT 数据。
- 还没有实现复习、测验、间隔重复和用户词汇水平诊断。
- 还没有支持 PDF、网页、Markdown、字幕等更多输入格式。
- 候选词评分仍是规则版，尚未基于真实用户学习数据优化。

## 下一步建议

下一阶段可以优先补三件事：

1. 替换完整 ECDICT、基础词表和目标词库，让候选词与释义质量更稳定。
2. 完善 DeepSeek 义项选择的评估集和失败回退策略。
3. 增加一个轻量前端，让用户能在页面上查看候选词并点击三种操作。
